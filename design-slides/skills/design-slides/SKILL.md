# Slide Designer

Generate polished, scrollable single-file HTML pages designed for screen-recording video content. Supports YouTube videos, Video Sales Letters (VSLs), and Webinar presentations.

The user typically has an outline (often generated by another skill) and wants it transformed into visuals they can present on camera.

## Video Type Detection

Determine the video type before proceeding. Use this decision tree:

```
What type of video content?
├── "YouTube" / "tutorial" / "educational" / "explainer" / outline-style content
│   → YouTube Video (read frameworks/youtube-video-framework.md)
│
├── "VSL" / "video sales letter" / "sales video" / persuasion-focused / selling a product
│   → Video Sales Letter (read frameworks/vsl-framework.md)
│
├── "webinar" / "perfect webinar" / "presentation" / "keynote" / long-form + selling
│   → Webinar Video (read frameworks/webinar-framework.md)
│
└── Ambiguous → Ask the user which type
```

Once you know the type, **read the corresponding framework file** before proceeding. The framework defines the section structure, ordering rules, and component affinities for that video type.

## Two-Phase Workflow

This skill operates in two phases because the user needs to approve the visual structure before you invest time building the full page.

### Phase 1: Visual Outline

When the user provides their content outline, **read the relevant framework file first**, then produce a **visual outline** — a structured plan that maps each section to specific visual components. Present it as a numbered list in the conversation:

```
VISUAL OUTLINE: [Title]
Video type: [YouTube / VSL / Webinar]
Estimated sections: [N] (for ~[X] min video)

1. [Section Title]
   Component: [component type from the library]
   Content summary: [what this section will show]

2. [Section Title]
   Component: [component type]
   Content summary: [what this section will show]

...
```

The framework file specifies section count guidance, required sections, and which components work best for each section type.

Wait for the user to confirm, adjust, or reorder before proceeding to Phase 2.

### Phase 2: Generate the HTML Page

Once the outline is approved, generate the complete single-file HTML page. Everything — CSS, any inline SVGs, layout — goes in one `.html` file. No external dependencies except Google Fonts (DM Sans) and the brand assets loaded via URL.

**CSS Strategy — Inline at Generation Time:**
1. Read `brand/brand.css` — this is the single source of truth for all component styles
2. Copy the entire contents of `brand.css` into a `<style>` block in the generated HTML file's `<head>`
3. Add any page-specific CSS overrides after the brand CSS within the same `<style>` block
4. The output file must be fully self-contained — no external CSS file references

This ensures output files are portable (single-file, no broken links) while all styles are maintained centrally in `brand.css`.

**Before generating:** Read `brand/brand.css` for all styles and `components/master-component-showcase.html` for the exact HTML structure of fixed sections and the slide scroll snap script.

#### File Naming Convention

```
~/Desktop/design-slides/[topic-slug]-[datetime].html
```

- **topic-slug:** lowercase, hyphenated topic name (e.g., `ai-agents-slides`, `ai-blueprint-vsl`, `scale-with-ai-webinar`)
- **datetime:** local machine time in the format `Mon-DD-YYYY-HH-MM-am/pm` (e.g., `Mar-15-2026-11-31-am`)

Create the `~/Desktop/design-slides/` directory if it doesn't already exist. This ensures the user can find the output file immediately on their Desktop.

#### Auto-Open in Browser

After saving the HTML file, automatically open it in the user's default browser:

- **macOS:** Run `open ~/Desktop/design-slides/[filename].html`
- **Windows:** Run `start "" "%USERPROFILE%\Desktop\design-slides\[filename].html"`
- **Linux:** Run `xdg-open ~/Desktop/design-slides/[filename].html`

Always attempt the open command. If it fails, tell the user the exact file path so they can open it manually.

## Screen-Recording Optimization

Every design decision serves one goal: making this page look great when scrolled through on camera at 1920x1080.

**Viewport and layout:**
- Design for exactly 1920x1080 viewport (16:9 recording)
- Max content width: 1200px, centered
- Each major section should fill roughly one viewport height so the presenter can pause on it
- Generous padding (40px+ on sides, 60px+ between sections)
- Smooth scroll behavior on the html element (`scroll-behavior: smooth`)

**Typography and readability on camera:**
- Font: DM Sans (loaded from Google Fonts)
- Heading weight: Semibold (600) — not bold or extra-bold
- Body weight: Regular (400)
- Base type scale (from brand): H1 50px, H2 42px, H3 30px, H4 23px, H5 19px, H6 16px, Body 16px, Small 14px
- For 1920x1080 screen recording, scale up for on-camera readability:
  - Hero title: 64–72px (scaled from H1)
  - Section headings: 40–48px (scaled from H2)
  - Card titles: 24–28px (scaled from H4)
  - Body text: 20–22px (scaled from Body)
  - Key stats or hero numbers: 60–88px
- Minimum text size anywhere on the page: 14px absolute floor — no exceptions
- Line height: 1.6+ for body text
- Text must be crisp against the light background — use black (#000) for primary text, muted tones for secondary

**Visual hierarchy for scanning:**
- Viewers are watching a video, not reading a document — they need to grasp each section in 3–5 seconds
- Lead every section with a bold visual element (diagram, stat, icon grid) before any paragraph text
- Use color accents (brand purple #4F30FE and light purple #DAB3FF) to draw the eye to key information
- Avoid walls of text — if a section has more than 3–4 sentences, break it into a visual component instead

## Brand Styling

Read `brand/brand.md` for the full brand specification. The essentials:

- **Theme:** Light mode — white background (`#ffffff`), dark text (`#000000`). Optional `.section-dark` class for accent sections.
- **Logo:** Displayed in the hero section, max height 42px, opacity 70%
- **Primary accent:** Purple (#4F30FE)
- **Secondary accent:** Light Purple (#DAB3FF)
- **Gradients:** Lavender-to-blue (`#DAB3FF → #262DFF`) for gradient text, button backgrounds, accent bars
- **Gradient text:** Used for highlight phrases in headings
- **Accent color text:** `#4F30FE` for single-color inline highlights
- **Supporting colors:** Hot pinks, bright greens — use sparingly
- **Font:** DM Sans — Semibold (600) for headings, Regular (400) for body
- **Buttons:** Primary CTA uses gradient background (`#DAB3FF → #262DFF`), white text, 10px radius
- **Border radius:** 12–16px for cards, 8–10px for smaller elements
- **Cards:** White bg (`#ffffff`), subtle shadow (`0 2px 12px rgba(0,0,0,0.04)`), light borders (`rgba(0,0,0,0.08)`)

### Combining Components

A single section on the page often combines 2 components — for example, a **callout box** followed by a **flow diagram**, or **stat cards** above a **card grid**. This is encouraged. The visual outline in Phase 1 should note when sections use combined components.

### Creating Custom Visuals

The component library covers the most common patterns, but you're not limited to it. If a concept needs a visual that doesn't fit any existing component — an unusual diagram, a custom SVG illustration, a unique layout — create it using inline SVG and CSS. The brand colors, typography, and spacing rules still apply.

### Enhanced Visuals with JS Libraries

For interactive or animated diagrams that go beyond static CSS/SVG, use these three libraries loaded from cdnjs.cloudflare.com:

**D3.js v7** — For custom data-driven diagrams: force-directed network graphs, Sankey flows, tree diagrams.
```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/d3/7.8.5/d3.min.js"></script>
```

**GSAP 3** — For scroll-triggered animations: counters that count up, elements that fade/slide in, staggered reveals.
```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/gsap/3.12.2/gsap.min.js"></script>
```

**Chart.js 4 (UMD build)** — For polished interactive charts. **Important: always use the UMD build** (`chart.umd.min.js`).
```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/Chart.js/4.4.0/chart.umd.min.js"></script>
```

Use these libraries selectively — not every page needs them. A page with 10 sections might use JS libraries in 1–3 of them for maximum impact.

### Hero (All Types)

The hero section has no background — it sits directly over the body background. Copy the structure from `templates/sections/hero.html` and adapt the content per the framework.

**HARD RULE:** The hero must include the logo, topic label, title with gradient highlight, subtitle, and 3 promise boxes. See the hero template for the exact HTML structure.

### About Me (YouTube: Required, Webinar: Optional, VSL: Never)

**HARD RULE: Always read `components/master-component-showcase.html` before generating any HTML file. Copy the About Me section (HTML + its CSS classes) exactly as it appears in that file — do not rewrite it from memory.**

The About Me section is identified by `<section class="about-me" id="about-me">` in the master file. Also available as a standalone extract in `templates/sections/about-me.html`.

### Next Step CTA (All Types — Content Varies)

**HARD RULE:** Copy the appropriate CTA section exactly from its template file:
- YouTube: `templates/sections/next-step-youtube.html`
- VSL: `templates/sections/next-step-vsl.html`
- Webinar: `templates/sections/next-step-webinar.html`

### Slide Scroll Snap (All Types)

**HARD RULE: Copy the slide scroll snap script exactly from `templates/sections/scroll-nav.html` — do not rewrite it from memory.**

Place it just before `</body>` in every generated HTML file.
